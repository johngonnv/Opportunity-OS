## Replit Message — Build 2-Layer Pipeline View Administration + Internal Business Admin Login

We need to build a **2-layer administration model for pipeline views** on top of the current universal Opportunity OS schema.

### Goal

Support:

* **Business-level control** of master pipeline templates
* **Client workspace admin control** of approved pipeline views inside their own workspace

This should **not** become a separate EMS-only CRM.
It should extend the current core architecture, which already supports universal CRM objects, multiple pipelines, opportunities, activities, tasks, organizations, contacts, and workspace structure. 

---

## Core Product Rules

* Do **NOT** let clients create pipeline views fully from scratch in v1
* **Business Admin** owns the master template library
* **Client Admin** only manages approved templates inside their workspace
* A workspace may support **multiple enabled pipeline views**
* For the current EMS client, prepare for multiple EMS views later, but enable only:

  * **Interfacility Transport**

Future EMS views may include:

* EMS Education
* Event Medical Staffing
* Air EMS

But do **not** build or enable those yet beyond template readiness.

# 1) Add Internal Business Admin Login / Access Layer

## Goal

We need a secure internal admin account type so our company can manage templates, publish pipeline views, and administer client workspaces without mixing this access with normal client workspace users.

## Build requirement

Extend the current authentication system with a **platform-level internal admin role**.

## Add user access types

Support these access levels:

* `business_super_admin`
* `internal_ops_admin`
* `client_workspace_admin`
* `standard_user`

## Recommended schema update for `users`

Add:

* `account_type` text default `'client_user'`
* `is_platform_admin` boolean default `false`
* `platform_role` text nullable

### `account_type` values

* `platform_user`
* `client_user`

### `platform_role` values

* `business_super_admin`
* `internal_ops_admin`

## Auth behavior

### Business Admin users

* Can log into an internal admin console
* Can access platform-level template and workspace controls
* Can manage pipeline view templates across all clients
* Can publish templates to workspaces
* Should **not** require normal membership in each workspace to administer templates

### Client Workspace Admin users

* Log into the normal client experience
* Can access only workspaces they belong to
* Can only manage approved pipeline views inside their workspace
* Cannot edit platform-level master templates

## Internal login UX

Create a distinct internal login route:

* `/admin/login`

After login, internal business admins land at:

* `/admin`

Do **not** mix this with the normal client workspace landing page.

# 2) Build Two Administration Layers

## Layer 1 — Business Admin Layer

### Purpose

Internal product administration for Opportunity OS

### Business Admin capabilities

* Create pipeline view templates
* Edit master template metadata
* Assign template to one or more verticals
* Mark template as:

  * active
  * inactive
  * locked
  * client-editable
* Define default:

  * probabilities
  * required fields
  * saved views
  * automation suggestions
  * dashboard widgets
* Publish template for workspace use
* Archive template
* Clone template
* Preview template before publishing

## Layer 2 — Client Workspace Admin Layer

### Purpose

Allow customer admins to manage which approved pipeline views are active in their workspace

### Client Admin capabilities

* View available published pipeline view templates allowed for their workspace
* Enable / disable a pipeline view in their workspace
* Set display order of enabled pipeline views
* Set default landing pipeline view
* Configure workspace-level saved views
* Configure user/team visibility for enabled pipeline views
* Configure optional field visibility
* NOT allowed to edit locked master template definitions

# 8) EMS Pipeline View to Build First

## Pipeline name

* **Interfacility Transport**

## Business context

This EMS client provides:

* BLS
* ALS
* CCT

For Nevada / Las Vegas Ground EMS:

* do **not** use traditional contracting language
* use:

  * service agreement
  * agreement alignment
  * protocol alignment
  * go-live readiness

This client can only pick up patients in the **City of Las Vegas jurisdiction**.

We need to track:

* facility pain points
* current transport volume
* payer mix
* current provider relied on

Important contact roles include:

* Charge Nurse
* Director of Nursing
* Case Management
* House Supervisor

---

## Pipeline stages

Seed these stages in order:

1. Target Identified
2. Facility Engaged
3. Director Engaged
4. Discovery
5. Agreement Alignment
6. Go-Live
7. Active Account
8. Expansion

## Recommended stage probabilities

* Target Identified = 5
* Facility Engaged = 15
* Director Engaged = 30
* Discovery = 45
* Agreement Alignment = 60
* Go-Live = 75
* Active Account = 90
* Expansion = 100

## Stage meanings

* **Target Identified**
  facility identified as potentially in-jurisdiction and worth outreach

* **Facility Engaged**
  initial outreach has occurred

* **Director Engaged**
  charge nurse, director of nursing, or equivalent operational stakeholder is engaged

* **Discovery**
  pain points, current transport volume, payer mix, and incumbent provider captured

* **Agreement Alignment**
  discharge workflow / call protocol / operational agreement being aligned

* **Go-Live**
  facility is ready to send transports and staff know the process

* **Active Account**
  account has demonstrated consistent transport usage

* **Expansion**
  volume is growing or additional units / departments / campuses are being targeted

# 11) Active Account Logic

Use **consistency**, not signature.

System should mark an account as eligible for **Active Account** when:

* `protocol_go_live_date` is populated
* and either:

  * average qualified transports per week is **>= 1 for 4 consecutive weeks**
  * or qualified transports in the last 30 days is **>= 4**

This is the recommended operational definition for Active Account.
