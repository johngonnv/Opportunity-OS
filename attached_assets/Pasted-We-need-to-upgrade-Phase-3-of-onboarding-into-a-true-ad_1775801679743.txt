We need to upgrade Phase 3 of onboarding into a true admin decision layer.

Goal:
Turn `/admin/onboarding/[id]/review` from a simple AI suggestion approval page into a structured, mobile-first review and provisioning control center.

This must align with Opportunity OS principles:
- AI may suggest, but cannot silently finalize high-impact decisions
- onboarding must seed actionable execution, not just classification
- workspaces should never feel blank after provisioning
- pipelines are template-driven and can be enabled without being visible
- master database growth and auditability must be preserved

Reference rules already established in the system:
- AI suggestions must be reviewable and logged
- AI cannot auto-assign parents, merge records, or close opportunities
- onboarding should capture industry, services, territory, and use case
- provisioning should enable pipelines, saved views, enrichment, and early opportunities

BUILD THIS AS FOLLOWS:

==================================================
1) PHASE 3 PURPOSE
==================================================

Phase 3 = Admin Review and Lock

This phase is the final human validation layer before provisioning.

It must let internal Opportunity OS admins:
- review each AI-generated recommendation block independently
- approve, edit, or reject each block
- compare suggested values vs final values
- block provisioning until all required sections are resolved
- preserve AI suggestion history for learning and audit

This phase should feel like:
"AI recommended the setup. Admin decides the truth."

==================================================

5) ITEM CARD UX RULES
==================================================

Each item card must show:
- label
- suggested value
- confidence badge
- current decision status
- actions

Per-item actions:
- Approve
- Edit
- Reject

Statuses:
- PENDING
- APPROVED
- EDITED
- REJECTED

Behavior:
- Approve = accept AI suggestion as final
- Edit = open inline modal or bottom sheet editor, save edited final value, mark EDITED
- Reject = require rejection reason, mark REJECTED
- Rejected required items must be fixed before provisioning
- Optional items may remain rejected if admin intentionally rejects them

For array items:
- admin can remove entries
- admin can reorder entries
- admin can add custom entries
- edited result becomes final value

Show:
- Suggested
- Final
- Source
- Confidence
- Last action by
- Last action at

7) NEW DATA MODEL
==================================================

Add 2 new tables:

A) `workspace_onboarding_sessions`
Purpose:
Store onboarding session state and final approved config before provisioning.

Recommended columns:
- id
- workspace_id nullable
- client_name
- source_intake_json jsonb

- required_section_count

- locked_by_user_id nullable
- locked_at nullable
- provisioned_at nullable
- created_at
- updated_at

B) `workspace_onboarding_review_items`
Purpose:
Store each reviewable recommendation block independently.

Recommended columns:

- session_id fk
- group_key
- item_key
- label
- suggested_value_json jsonb
- final_value_json jsonb nullable
- source_json jsonb nullable
- confidence_band enum('HIGH','MEDIUM','LOW')

- sort_order integer
- reviewed_by_user_id nullable
- reviewed_at nullable
- created_at
- updated_at

Important:
- Keep both suggested_value_json and final_value_json
- final_value_json is the source of truth for provisioning
- if approved with no edits, final_value_json should copy suggested_value_json
- every change must be auditable

9) API ROUTES
==================================================

Add admin routes under:
- `/api/admin/onboarding`

Required routes:

Session routes
- `GET /admin/onboarding/:id`
  - return session summary + grouped review items + progress
- `POST /admin/onboarding`
  - create new onboarding session from intake + ai recommendations
- `POST /admin/onboarding/:id/lock`
  - validate session completeness and mark READY_TO_PROVISION if valid
- `POST /admin/onboarding/:id/provision`
  - provision workspace from final reviewed values

Review item routes
- `POST /admin/onboarding/:id/items/:itemId/approve`
- `POST /admin/onboarding/:id/items/:itemId/edit`
- `POST /admin/onboarding/:id/items/:itemId/reject`

Utility routes
- `GET /admin/onboarding/:id/progress`
- `POST /admin/onboarding/:id/rebuild-items`
  - rebuild review items from latest AI recommendation if session not yet locked

Behavior rules:
- approve writes final_value_json = suggested_value_json
- edit writes final_value_json = edited payload and marks EDITED
- reject requires rejection_reason
- progress endpoint returns required/resolved/blocking items
- provision route hard-blocks if required items unresolved
