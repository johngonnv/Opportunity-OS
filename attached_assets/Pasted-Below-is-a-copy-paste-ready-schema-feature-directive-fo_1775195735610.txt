⸻

Build the Master Organization Intelligence Layer as a first-class platform feature for Opportunity OS.

Goal

Create a company-owned, cross-workspace master organization graph that becomes the primary intelligence layer for:
	•	healthcare enterprise / regional / facility ownership mapping
	•	government contracting parent / subsidiary / affiliate mapping
	•	duplicate prevention
	•	workspace-to-master linking
	•	sibling discovery
	•	enterprise-level selling strategy

Do not create separate master databases for healthcare and GovCon.
Use one shared master database with vertical overlays.

The master database should sit above all workspaces and be managed at the platform-admin level.

⸻

1. Core Architecture

Layer A — Master Organization Graph

Platform-owned, cross-workspace, canonical intelligence database.

Layer B — Vertical Overlays

Industry-specific fields for healthcare, GovCon, and future business verticals.

Layer C — Workspace Organizations

Client/workspace-level organizations used for execution inside each workspace.

Workspace organizations should link to master organizations using:
	•	master_organization_id

The master organization graph is the reusable source of truth for identity, hierarchy, aliases, and ownership structure.

4. Normalization Requirements

Name normalization

At insert time:
	•	lowercase
	•	strip punctuation where appropriate
	•	remove legal suffixes like inc, llc, corp, corporation, ltd, co
	•	collapse whitespace

Domain normalization

At insert time:
	•	strip protocol
	•	strip www
	•	strip path/query
	•	store root domain only

Matching

Use PostgreSQL pg_trgm for fuzzy similarity on:
	•	normalized_name
	•	normalized_alias_name

Do not load all orgs into memory for similarity matching.

5. Core Feature Set

Feature A — Master Linking Engine

When a workspace org exists without master_organization_id, generate master-match suggestions.

Inputs:
	•	org name
	•	normalized name
	•	domain
	•	aliases
	•	industry
	•	place data if available

Actions:
	•	suggest top 3 master matches
	•	allow quick link
	•	allow manual search
	•	allow promote to new master org

⸻

Feature B — Structure Scan

From an organization card, run:
	1.	master database lookup
	2.	external validation if needed
	3.	LLM synthesis only if ambiguity exists
	4.	human approval
	5.	optional writeback to master graph

This supports parent detection, enterprise mapping, and sibling discovery.

⸻

Feature C — Master Promotion Workflow

Allow approved workspace organizations to be promoted into the master graph.

Rules:
	•	platform-aware
	•	explicit confirmation required
	•	optional toggle:
	•	“Add to shared knowledge base”

Defaults:
	•	internal users: ON
	•	client users: OFF

⸻

Feature D — Ultimate Parent Resolver

After parent is identified:
	•	recursively find highest parent
	•	save as suggested_ultimate_parent_name
	•	support enterprise rollups

⸻

Feature E — Sibling Discovery

For any org with a parent:
	•	return sibling orgs under same parent
	•	exclude current org

Expose in UI:
	•	“Other facilities in this system”
	•	“Other entities under this enterprise”

⸻

Feature F — Master Coverage Metrics

Track by workspace:
	•	total orgs
	•	linked orgs
	•	unlinked orgs
	•	coverage percentage

Expose these in admin.

8. LLM Usage Rules

Use GPT-4o mini only for reasoning/synthesis.

Trigger only when:
	•	internal confidence < 0.60
	•	and external candidates exist
	•	and results are ambiguous/conflicting

Do not call LLM when:
	•	master DB returns strong match
	•	deterministic signals are already strong
	•	no external data exists

LLM is an interpretation layer only, not a system of record.
